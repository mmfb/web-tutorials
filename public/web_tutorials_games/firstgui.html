1- Change the index.html in the public directory to be our game page

Normally we would create a simple main page in index.html that introduced the game and with a link or button to the game itself. For now we will skip this and go directly to our game page, since it is the main objective of our project.

HTML files define the static content of a web page and the grouping of that content into an organized document.

HTML means HyperText Markup Language. HyperText means we have hyperlinks that may connect parts of the text to other pages or other parts of the same page. Markup Language means there is a markup syntax that defines types of contents and organizes the document

IMPORTANT: HTML does not define how content will show visually, that will be done using CSS. However, browsers define some base CSS rules for many HTML elements and so you will still have a basic formatting even before making your CSS.

You can see more about HTML here but we can list some of the elementsyou can use:

  • Text: From simple paragraphs (p element) to several title texts of increasing importance (h1 is the most important, h2 is right next and so on)
  • Images (img element): You can link image files from inside your server of outside images using their Url
  • Links (a element): Links are extremely important since they allow our pages to be Hypertext instead of simple text, meaning you can link to other pages or to other parts of your page using links
  • Tables (table, tr and td elements) and lists (ul, ol and li elements): Make sure the content you want to show is a table or list. Do not use table just to align something visually, that will be done with CSS grid or flex (with advantages, like allows us to change the layout to allow for mobiles, for instance)
  • Layout elements (header, nav, footer, aside, section, article, figure/figcaption elements ): These elements are extremly important since they define the structure of our web page. Nothing worse in the current days than a page that only uses divs for everything. The layout elements are close to how information is shown for instance in a newspaper or magazine, defining clear layout elements that have specific meaning. For instance search engines like google know that a Header is more important than a section, and CSS can also use these tags to create rules that apply to many layouts. You can see a basic introduction to these elements here.
  • Since HTML files are what is called to show a page, Html also have links to CSS (link element) and Javascrit (script element) files.

    • Stage all files to be commited using the + sign that appears when you mouseover Changes.

    • When all is commited a button should show for you to create a repository on GitHub. When pressed it will:

2- Define your project and REST Api

  • First define what will be your project (if you have not already) in general terms
  • Now see what functionalities you would like to have
  • Identify what entities you will need. From those and their relations you will start creating the database. From those, the database and the functionalities you will determine the rest resources you need
  • Start defining endpoints you will need for the functionalities you defined and define to what resource they correspond
  • Divide the work between all collaborators and start creating the documentantion as we talked about in the rest interface tutorial

3- Create your code

Since you are learning, this step and step 2 will iterate together until you reach the goal. You will create some rules and try to implement, and then go back to change something because you found out there is a better way, you need to add something more or the project idea changed.

Divide the tasks by all collaborators. Does not mean you cannot help each other or work together, but you each have your own tasks that you are responsible for, and can work by yourself also. Some tasks may be done has a group.

Do not forget to test:

  • Test the database queries on the database client first (pgadmin)
  • Test the server endpoints individually using the browser (get) or postman
  • The client side (when you start that part) is the last thing to test, since it depends on all others

Also be carefull with:

  • The paths of the endpoints and methods used. Follow what was teached in class to define a REST interface that follows a clear template
  • Names. Always use the same naming convention for Routes and Models files, for variables, for function names
  • Always set the try/catch and always place some console.log on the catch to print the error
  • Always write something on the beggining of the route code to see that you got to that endpoint and what was sent
  • If you are having a hard time debugging start placing some console.log in the code to see what is happening (don't forget to delete them after if they are too many)

The basics of creating a rest api were described in the rest interface tutorial but you will learn more techniques on the course, make your own research and combine all those to create specific solutions. You will also learn how to create the client-side code during the course (first try here).
